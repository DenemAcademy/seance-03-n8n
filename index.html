Séance 03 n8n MCP + API

n8n, workflows, nodes, API et MCP

Dans cette séance, on part de zéro : ce qu’est n8n, pourquoi on l’utilise quand un client veut une automatisation visuelle, comment lire les nodes, comment comprendre le JSON, puis comment connecter Claude Code et Codex avec MCP ou avec l’API n8n.

Le point important : n8n reste un outil dense. L’objectif n’est pas de tout connaître par cœur dès maintenant. L’objectif est de comprendre assez bien le fonctionnement pour demander ensuite à Codex ou Claude Code de créer, corriger et expliquer un workflow propre.

Fil rouge

Une demande client arrive depuis un formulaire. n8n la vérifie, prépare une réponse, alerte l’équipe, enregistre une trace, puis laisse une execution lisible pour vérifier.

1 · Comprendre

Pourquoi n8n sert à montrer un processus client étape par étape.

2 · Nodes

Trigger, action, logique, IA, erreur : chaque node a un rôle simple.

3 · JSON

Lire les champs qui entrent et sortent avant de corriger un workflow.

4 · Agents

API pour créer/modifier, MCP pour utiliser des outils exposés.

Comprendre 01 Source

n8n en phrase simple

n8n est un outil d’automatisation visuelle. Il relie des services entre eux : formulaire, email, Slack, Notion, CRM, API, IA. Une action arrive, n8n la reçoit, applique une logique, puis déclenche la suite. C’est utile quand un client veut une automatisation visible, contrôlable et modifiable sans cacher tout le travail dans du code.

Ce que vous faites

Retenez une phrase courte : `n8n fait circuler une donnée entre plusieurs outils, avec des règles visibles.` Cette phrase suffit pour commencer. Si la définition devient trop technique, la pratique devient floue.

Tip terrain

Ne commencez pas par chercher le node parfait. Commencez par écrire le geste manuel que vous voulez supprimer.

Comment vérifier

Vous savez expliquer la différence entre une discussion avec une IA et un workflow qui se déclenche tout seul.

Objectif = automatiser une suite d’actions vérifiables

Automatiser ne veut pas dire tout rendre automatique. Le but de la séance est donc de comprendre le chemin : déclencheur, donnée, nodes, sortie, preuve.

Comprendre 05 Source

La donnée circule en JSON

n8n transporte des données sous forme de JSON. Dit simplement, le JSON est une façon de ranger des informations avec des noms de champs et des valeurs. Ce n’est pas juste du texte sur un écran. Un node reçoit des champs, les modifie, puis transmet une sortie. Comprendre ça évite de chercher une erreur au mauvais endroit.

Ce que vous faites

Quand un node se termine, ouvrez sa sortie et repérez les noms exacts des champs : `name`, `email`, `message`, `date`. Le node suivant doit utiliser ces noms exacts. Codex ou Claude Code peut aider à écrire le JSON, mais vous devez savoir le relire.

Comment vérifier

Vous savez retrouver un champ dans la sortie d’un node au lieu de l’inventer.

Question à se poser

Quel champ précis le node suivant doit-il lire ?

À copier / adapter 
{{ $json.email }}
{{ $json.message }}
{{ $json.date }}

Tip terrain

Si un champ est vide, ne corrigez pas le node final. Remontez au premier node où le champ disparaît.

Comprendre 06 Source

Cloud ou self-hosted

n8n peut être utilisé en cloud ou hébergé soi-même. Le cloud simplifie le démarrage, les mises à jour et l’accès public aux webhooks. Le self-hosted donne plus de contrôle, mais demande plus de responsabilité technique : serveur, variables, sauvegardes, sécurité et mise à jour.

Ce que vous faites

Pour une première séance, partez sur l’idée simple : n8n Cloud pour apprendre vite, self-hosted quand l’équipe sait gérer l’infrastructure. Le choix technique ne doit pas bloquer la compréhension du workflow.

Tip terrain

Comment vérifier

Cloud = plus simple pour démarrer
Self-hosted = plus de contrôle, plus d’entretien

API, node et MCP

Trois mots reviennent souvent. Un node est une brique n8n prête à configurer. Une API est une porte HTTP pour parler à un service. MCP est un protocole qui permet à un agent comme Claude Code ou Codex d’utiliser des outils de manière structurée.

Ce que vous faites

Gardez cette règle : node quand l’intégration existe, HTTP Request quand il faut appeler une API, MCP quand un agent doit découvrir et utiliser des outils.

Comment vérifier

Vous savez choisir entre node prêt à l’emploi, HTTP Request et MCP sans mélanger les rôles.

Question à se poser

Est-ce que je connecte deux applications, ou est-ce que je donne un outil à un agent ?

Test et rollback

Un agent peut aller vite. La sécurité vient du rollback. Export JSON, copie de workflow, désactivation facile, test manuel, trace d’execution. Sans ça, une automatisation peut casser silencieusement.

Ce que vous faites

Avant chaque changement API, exportez ou copiez. Après changement, lancez un test manuel. Si le résultat est mauvais, revenez à la copie.

Comment vérifier

Vous pouvez annuler le changement en moins d’une minute.

Gestion des secrets

Les secrets sont souvent la partie la plus fragile : tokens, API keys, OAuth, webhooks privés. Une fuite peut donner accès à des outils réels. Le support et GitHub ne doivent jamais contenir ces valeurs.

Ce que vous faites

Stockez les secrets dans credentials, variables d’environnement ou gestionnaire de secrets. Remplacez toute valeur dans les exemples par `<TOKEN>`.

Comment vérifier

Un `rg` sur les fichiers ne trouve aucune clé réelle.
